# 毛毛虫绘本/音频管理 App（安卓）需求方案（PRD）

## 1. 背景与问题
毛毛虫点读笔的常见资源与玩法：
- 点读包：`.bnl`，通常放入点读笔的 `BOOK/` 目录（不同型号目录可能不同）。
- 音频：故事/儿歌等，常放入 `MP3/` 目录播放。
- 点读贴/录音贴：需要将音频放入 `DIYAUDIO/`（或同类目录），并按 `REC+贴纸编码` 命名（例：`REC1205.mp3`），点读笔点对应贴纸即可播放。

用户痛点（来自实际使用反馈）：
- 资源整理成本高：文件多、命名规则强、容易重复/冲突。
- 增量更新困难：新增少量音频，却需要反复整理/重复处理整包文件。
- 下载/同步体验差：部分型号依赖云盘下载，稳定性与速度不可控。

## 2. 产品目标
做一款安卓 App，用于“在手机上整理资源并生成可直接拷贝的目录结构”，核心能力：
1) 对指定文件夹批量改名（按点读贴、排序编号、正则替换等规则）。
2) 以“绘本”为中心管理资源（点读包、音频、点读贴映射），支持增量更新与导出。
3) 通过安卓系统的文件访问框架（SAF）操作用户选定的目录（手机存储、U盘/OTG、部分可写的外接存储）。

非目标（MVP 不做）：
- 直接对接官方云盘/官方服务下载（需要协议与账号体系，且不稳定因素多）。
- 直接“远程下发到点读笔”（取决于硬件协议；MVP 仅做本地文件准备与导出）。

## 3. 目标用户与使用场景
- 家长：管理绘本点读包、做点读贴、整理故事音频，频繁“新增少量内容”。
- 线下场景：外出离线播放，提前把音频整理到可拷贝的目录。

典型场景：
- 场景 A（点读贴）：选中一组音频 → 自动生成 `RECxxxx` 命名 → 导出到 `DIYAUDIO/`。
- 场景 B（故事机）：将一批音频按播放顺序改成 `0001_*.mp3` → 导出到 `MP3/`。
- 场景 C（点读包）：整理 `.bnl` 文件命名 → 归档/导出到 `BOOK/`。

## 4. MVP 范围（第一期）
### 4.1 核心功能
1) 文件夹授权与浏览（SAF）
   - 用户选择一个“工作目录”（可重复进入、持久化授权）。
   - 展示目录下文件列表（名称、扩展名、大小、修改时间）。

2) 批量重命名（自动修改名称）
   - 预览改名结果：旧名 → 新名；检测冲突（重名、非法字符、长度）。
   - 一键执行：逐个重命名；记录成功/失败原因；支持“失败不阻塞”（继续处理后续文件）。

3) 改名规则（MVP 提供 3 种）
   - 点读贴规则：`REC{编码}`（编码可输入起始值并按排序递增；或手工对单个文件指定编码）。
   - 顺序编号规则：`{index:04d}_{原名}`（按文件名排序/修改时间排序/自定义排序）。
   - 正则替换规则：`pattern -> replacement`（仅对文件名主体处理，保留扩展名）。

4) 绘本资源“项目化”管理（轻量）
   - 创建“绘本项目”：书名、备注、封面（可选）、资源目录引用。
   - 给项目添加资源：点读包（`.bnl`）、音频集合、点读贴映射（音频 ↔ 贴纸编码）。
   - 导出结构：生成（或复制到）`BOOK/`、`MP3/`、`DIYAUDIO/` 三类目录（MVP 先做“生成目标目录与改名”，复制/移动作为下一期增强）。

### 4.2 体验要求
 - 全程离线可用。
 - 执行改名之前必须有“预览 + 冲突提示”。
 - 执行过程可取消（已完成的不回滚；后续版本再做事务性回滚）。

## 5. 第二期（增强方向）
- 增量同步：基于文件指纹（size+mtime+hash）判断新增/变更，仅处理差异。
- 复制/移动与“导出包”：一键把整理后的资源复制到目标目录（OTG/U盘/外接存储）。
- 导入/导出映射：CSV/JSON（例如贴纸编号与音频的对应表）。
- 音频工具：格式转换/音量归一化/切分（需要本地 FFmpeg 等能力与性能评估）。

## 6. 约束与风险
- 不同型号点读笔目录结构可能不同（`BOOK/MP3/DIYAUDIO` 仅为常见命名）；App 需允许用户自定义目标目录名。
- Android 对外接存储写入能力受系统/厂商限制；SAF 可覆盖大部分“用户明确选择的目录”场景，但不保证所有设备/文件系统均可写。
- 批量改名存在冲突与不可逆风险；必须提供预览与冲突策略（跳过/自动加后缀）。

## 7. 验收标准（MVP）
- 用户能选择任意文件夹并列出文件。
- 任一改名规则都能在“预览列表”中正确展示改名计划，并能检测到冲突。
- 点击执行后，实际文件名与预览一致；失败的文件给出明确错误信息。
- “点读贴规则”可用起始编码自动生成 `RECxxxx`，并保留扩展名。
